# Meal Prep Planner — поэтапное ТЗ для Claude Code

## Как работать с этим документом

ТЗ разбито на **6 этапов**. Claude Code должен идти по ним строго по порядку, **не перескакивая**. На каждом этапе:

1. Сообщить, что сделано на предыдущем этапе (если он был)
2. Описать, что планирует сделать на текущем этапе
3. Дождаться явного подтверждения от пользователя («ок», «продолжай», «да»)
4. Только после подтверждения начать работу
5. По завершении этапа — отчитаться: что сделано, какие файлы созданы/изменены, что нужно проверить

**Запрещено:**
- Переходить к следующему этапу без подтверждения
- Делать задачи из других этапов «попутно»
- Принимать решения о расширении объёма работ без согласования

**Если возникает вопрос или неоднозначность** — задать его пользователю, не делать на своё усмотрение.

## Контекст проекта

Превращаем существующий прототип `meal-prep-planner.html` (однофайловое приложение на ванильном JS с localStorage) в **Telegram Mini App** с:
- Облачным хранением (Supabase)
- Авторизацией через Telegram
- Хостингом на Vercel
- Новой дизайн-системой (Gilroy + розово-малиновая палитра)

Прикреплённые файлы:
- `meal-prep-planner.html` — текущий код
- `Gilroy-Light.woff2`, `Gilroy-ExtraBold.woff2` — шрифты

Пользователь по запросу предоставит: Supabase URL, anon key, Bot Token.

## Этап 1. Структура данных и Supabase SQL

### Что сделать

**1.1.** Подготовь SQL-скрипт `supabase/schema.sql` для создания таблиц:

- **`users`**: `telegram_id` (bigint, PK), `username` (text), `first_name` (text), `created_at` (timestamp)
- **`recipes`**: `id` (uuid, PK), `user_id` (bigint, FK к users.telegram_id, nullable для дефолтных), `name`, `type`, `category`, `emoji`, `recipe`, `serve`, `time` (int), `meat` (jsonb), `ingredients` (jsonb), `nutrition` (jsonb, nullable), `is_default` (boolean), `created_at`
- **`plan`**: `id` (uuid, PK), `user_id` (bigint, FK), `day_index` (int 0-13), `recipe_id` (uuid, nullable, FK к recipes), `side` (text, nullable), `status` (text), `week_start` (date), `updated_at`
- **`active_recipes`**: `user_id` (bigint), `recipe_id` (uuid), composite PK
- **`sides`**: `id` (uuid, PK), `name`, `emoji`, `portion` (int), `nutrition` (jsonb), `is_default` (boolean) — справочник гарниров

**1.2.** Включи Row Level Security на всех таблицах с пользовательскими данными. Политика: пользователь видит только свои строки (по `user_id` = `telegram_id` из JWT claim).

**1.3.** Подготовь SQL-сидер `supabase/seed.sql` — вставка дефолтных рецептов из `DEFAULT_RECIPES` (массив в текущем HTML) с `is_default = true` и `user_id = NULL`. Также вставка дефолтных гарниров из массива `SIDES` с КБЖУ для стандартных порций.

**1.4.** Подготовь функцию `copy_defaults_to_user(p_user_id bigint)` — копирует все `is_default = true` рецепты в данные пользователя.

### Что НЕ делать на этом этапе

- Не писать Edge Function (это Этап 2)
- Не трогать HTML
- Не подключать Supabase к фронту

### По завершении

Отчитайся:
- Какие файлы созданы (`supabase/schema.sql`, `supabase/seed.sql`)
- Какие таблицы будут созданы и с какими полями
- Какие RLS-политики настроены
- Дай инструкцию: «Откройте Supabase → SQL Editor → запустите сначала `schema.sql`, потом `seed.sql`. Сообщите, когда выполнено успешно.»

**Дождись подтверждения от пользователя, что SQL выполнен в Supabase. Только потом переходи к Этапу 2.**

## Этап 2. Edge Function авторизации

### Что сделать

**2.1.** Создай файл `supabase/functions/auth-telegram/index.ts`:

- Принимает POST с `{ initData: string }` от клиента
- Проверяет подпись Telegram через HMAC-SHA256 с использованием `BOT_TOKEN` из env
- Если валидна — апсертит запись в таблицу `users`
- Если запись новая — вызывает `copy_defaults_to_user(telegram_id)`
- Генерирует JWT с claim `telegram_id` и `role: 'authenticated'` (для совместимости с Supabase RLS)
- Возвращает `{ token: string, user: {...} }`

**2.2.** Создай README в `supabase/functions/auth-telegram/README.md`:
- Команды для деплоя через Supabase CLI
- Какие переменные окружения нужны (`BOT_TOKEN`)
- Пример запроса для теста

### Что НЕ делать на этом этапе

- Не подключать Edge Function к фронту (это Этап 3)
- Не менять HTML

### По завершении

Отчитайся:
- Какие файлы созданы
- Дай пошаговую инструкцию пользователю:
  1. Установить Supabase CLI (если ещё нет): команды для Mac/Windows
  2. Залогиниться: `supabase login`
  3. Связать с проектом: `supabase link --project-ref <id>`
  4. Добавить секрет: `supabase secrets set BOT_TOKEN=<токен от @BotFather>`
  5. Задеплоить: `supabase functions deploy auth-telegram`
  6. Проверить через curl-запрос (дай готовую команду с заглушкой initData)

**Дождись подтверждения, что Edge Function задеплоена и отвечает. Только потом переходи к Этапу 3.**

## Этап 3. Подключение Telegram SDK и авторизация на фронте

### Что сделать

**3.1.** В `meal-prep-planner.html` в `<head>` добавь подключение Telegram SDK:
```html
<script src="https://telegram.org/js/telegram-web-app.js"></script>
```

**3.2.** Создай в начале `<script>` секцию `CONFIG`:
```js
const CONFIG = {
  SUPABASE_URL: 'ЗАМЕНИТЬ',
  SUPABASE_ANON_KEY: 'ЗАМЕНИТЬ',
  AUTH_FUNCTION_URL: 'ЗАМЕНИТЬ' // URL Edge Function
};
```

**3.3.** Добавь Supabase JS SDK через esm.sh (без бандлера):
```js
import { createClient } from 'https://esm.sh/@supabase/supabase-js@2'
```

**3.4.** Реализуй функцию `initApp()`:
- Получает `tg = window.Telegram.WebApp`
- Вызывает `tg.ready()` и `tg.expand()`
- Если `tg.initData` пустой → показывает заглушку «Откройте через Telegram-бота»
- Иначе — отправляет `initData` на Edge Function, получает JWT
- Создаёт supabase-клиент с этим JWT
- Сохраняет JWT и `tg.initDataUnsafe.user` в переменные модуля (НЕ в localStorage)
- Показывает индикатор загрузки на время авторизации

**3.5.** На время этого этапа: **не трогай существующую логику работы с localStorage**. Просто добавь авторизацию параллельно. Если авторизация прошла — выводим в консоль `console.log('authenticated', user)`. Замена localStorage будет на Этапе 4.

### Что НЕ делать на этом этапе

- Не убирать localStorage
- Не менять дизайн
- Не добавлять новые экраны

### По завершении

Отчитайся:
- Какие изменения сделаны в HTML

## Этап 6. Новая функциональность: «Сегодня», жизненный цикл, КБЖУ

### Что сделать

**6.1. Контекстный стартовый экран.**
- При запуске определи день недели: `new Date().getDay()` (0 = воскресенье)
- Воскресенье → стартовая вкладка «План»
- Будний день → новая вкладка «Сегодня»
- Добавь поле `state.weekStart` — дата начала текущего цикла планирования

**6.2. Вкладка «Сегодня».**
- Дата и день недели крупно
- Карточка ужина: название + гарнир + КБЖУ компактно
- Статус заготовки крупно (иконка + текст)
- Контекстное действие — через **нативную Telegram MainButton**:
  - `prepped` → текст «Достать из морозилки»
  - `thawing` → «Готовить»
  - готовится → «Съедено»
  - `null` → «Запланировать неделю» (переход на «План»)
- Внизу мелким текстом — что на завтра

**6.3. Жизненный цикл заготовки.**
- В режиме «План» (воскресенье): кнопка «Заготовил» рядом с каждым активным рецептом → меняет статус всех его дней на `prepped`
- В «Сегодня»: одна кнопка двигает статус: `prepped` → `thawing` → `done`
- В списке дней в режиме «План»: иконка статуса каждого дня (🧊 / 💧 / ✓)

**6.4. КБЖУ.**
- Опциональное поле `nutrition` в рецептах: `{ kcal, protein, fat, carbs }`
- Поля `nutrition` и `portion` (граммы) для гарниров (уже в Supabase)
- Карточка рецепта: компактная плашка с КБЖУ
- Если КБЖУ нет — плашку не показываем
- Форма добавления рецепта: 4 опциональных поля
- В режиме «План»: блок «Среднее за ужин недели» — считается по дням с полными данными
- Никаких подсветок норм. Только цифры.

**6.5. Граничные случаи:**
- Будни, на сегодня нет плана → CTA «Запланируй неделю»
- Воскресенье, остались `prepped` от прошлого цикла → подтверждение «Начать новую неделю?»
- Воскресенье, план пустой → сразу режим планирования

**6.6. Telegram MainButton:**
- При смене вкладки — обновлять текст и обработчик
- На вкладках без основного действия — `tg.MainButton.hide()`
- Не дублировать кнопку в HTML, если она вынесена в MainButton

### Что НЕ делать на этом этапе

- Не добавлять то, чего нет в этом списке (бюджет, push, расширение белков и т.д.)
- Не менять дизайн-систему

### По завершении

Отчитайся:
- Что добавлено
- Как протестировать каждый сценарий:
  1. Открыть в воскресенье — должна открыться вкладка «План»
  2. Открыть в будни с заполненным планом — вкладка «Сегодня»
  3. Открыть в будни с пустым планом — CTA на планирование
  4. Полный цикл: запланировать → отметить «Заготовил» → утром «Достать» → вечером «Готовить» → «Съедено»
  5. Проверить КБЖУ-итог на «Плане»
- Финальная инструкция: как залить в GitHub, как подключить Vercel, как привязать к боту в @BotFather
